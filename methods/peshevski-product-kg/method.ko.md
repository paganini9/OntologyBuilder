# AI Agent 기반 제품 지식그래프 (3-에이전트)

> 출처: Peshevski, Stojanov, Trajanov, *An AI Agent-Driven Framework for Automated Product Knowledge Graph Construction in E-Commerce*, 1st GOBLIN Workshop 2025 (arXiv:2511.11017).

## 1. 한 줄 요약
사전 정의된 스키마나 수작업 규칙 없이, **3개의 전담 LLM 에이전트**(① 온톨로지 생성·확장 → ② 온톨로지 정제 → ③ KG 개체 채우기)로 제품 설명 텍스트에서 제품 지식그래프를 처음부터 자동 구축한다. 본 프로젝트(제품개발 온톨로지)와 가장 가까운 선례.

## 2. 핵심 개념
- **온톨로지 우선(ontology-first)**: SAC-KG류가 트리플부터 뽑는 것과 달리, **스키마(클래스·속성)를 먼저 짓고** 그 틀에 개체를 채운다.
- **에이전트 분업**:
  - **① 생성·확장 에이전트**: 제품 설명에서 클래스(제품 유형·부품)와 속성(스펙·관계)을 발견해 온톨로지를 키운다.
  - **② 정제 에이전트**: 중복 클래스 병합, 이름 표준화, 속성 정리(SAC-KG의 verifier+pruner 역할에 해당).
  - **③ 개체 채우기 에이전트**: 정제된 스키마에 따라 각 제품을 개체(individual)로 인스턴스화하고 속성값을 채운다.
- **평가 지표**: 정밀도보다 **property coverage**(속성 충족률) 중심. 원논문은 에어컨 제품 설명에서 97%+ coverage 보고.

## 3. 구축 프로세스 (단계별)
1. **제품 설명 수집** — `product_descriptions.txt`, 한 줄(또는 한 문단)에 제품 1개 설명.
2. **① 온톨로지 생성·확장 (제품마다 반복)** — 각 설명에서 클래스·객체속성·데이터속성을 추출해 누적 스키마에 더한다.
3. **② 온톨로지 정제** — 누적 스키마에서 유사·중복 클래스를 병합하고 이름을 표준화한다(단수/복수 정규화 포함).
4. **③ KG 개체 채우기** — 각 제품을 정제된 스키마의 클래스로 인스턴스화(개체 노드 생성)하고, 추출된 스펙을 데이터속성 값으로, 부품/관계를 객체속성으로 연결한다.
5. **산출** — 단계(① 제품별 스키마 확장 → ② 정제 → ③ 개체 채우기)마다 스냅샷을 남겨 "스키마가 서고 개체가 채워지는 과정"을 보여주고, `ontology.ttl`(스키마+개체), `ontology.json`(그래프), `steps.json`을 쓴다.

## 4. 입력 / 출력
| 구분 | 파일 | 설명 |
|------|------|------|
| 입력 | `product_descriptions.txt` | 제품 설명(한 줄/문단당 제품 1개) |
| 출력 | `ontology.ttl` | 스키마(클래스·속성) + 개체(individual) OWL (Turtle) |
| 출력 | `ontology.json` | Cytoscape nodes(클래스·개체)/edges(관계) |
| 출력 | `steps.json` | 단계별 스냅샷 — 생성·정제·채우기 재생용 |

## 5. LLM 백엔드
- 기본 `mock`: 키 없이 결정론적으로 동작. 생성은 설명에서 대문자 명사→클래스, 스펙 키워드→데이터속성, 관계 동사→객체속성으로 추출; 정제는 결정론 규칙 병합; 개체는 제품별로 `Product_1` 식 ID로 인스턴스화.
- `gemini`/`anthropic`(api): 키가 있으면 3개 에이전트를 실제 LLM으로 수행해 더 풍부한 스키마·개체를 만든다. 키 없으면 자동 MOCK 폴백.

## 6. 직접 해보기
1. `samples/product_descriptions.txt` 를 자신의 제품 카탈로그·스펙 설명으로 교체.
2. 사이트의 **Run** 버튼(또는 `python pipeline.py samples runs/out`) 실행.
3. 단계 슬라이더로 ① 스키마 확장 → ② 정제 → ③ 개체 채우기 흐름을 확인.
