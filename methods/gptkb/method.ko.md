# GPTKB — LLM의 파라미터 지식을 추출해 지식베이스로 구체화

> 출처: Hu, Nguyen, Ghosh, Razniewski, *GPTKB: Enabling LLM Knowledge Analysis via Extensive Materialization*, ACL 2025 (arXiv:2411.04920), [gptkb.org](https://gptkb.org).

## 1. 한 줄 요약
**입력 코퍼스 없이**, 시드 엔티티 몇 개에서 출발해 LLM에게 "이 엔티티에 대해 무엇을 아는가?"를 **재귀적으로 질의**하여, 모델 자체의 (파라미터에 저장된) 지식을 트리플로 추출하고 새로 발견된 엔티티를 재귀 확장해 지식베이스를 만든다.

## 2. 핵심 아이디어
- **입력 코퍼스 없음**: 읽을 문서 집합이 없다. 지식은 전적으로 모델 자체에서 나온다("코퍼스가 곧 모델"). 사용자는 시드 엔티티만 제공한다.
- **재귀적 시드 확장**: 각 엔티티에 대해 LLM에게 질의하고, 응답을 `(주어, 관계, 목적어)` 트리플로 파싱한다. 목적어가 다시 엔티티이면 새 노드로 등록되어 다음 확장 대상이 된다.
- **깊이 제한(depth limit)**: 너비 우선(BFS)으로 고정 깊이(여기서는 depth 2)까지만 확장해 무한 확장 없이 종료한다.
- **통합/중복 제거(consolidation/dedup)**: 여러 트리플이 같은 엔티티를 가리켜도 그 엔티티는 한 번만 질의한다(`visited` 집합). 따라서 구체화된 KB에 중복 노드가 없다.

## 3. 구축 과정 (단계별)
1. **시드** — `seed_entities.txt`에 시드 엔티티를 한 줄에 하나씩. (코퍼스 없음.)
2. **모델 질의(루프)** — BFS 큐에서 다음 엔티티를 꺼내, LLM(또는 MOCK 지식 테이블)이 JSON `{entity, facts:[{relation, object}]}`를 반환한다.
3. **구체화 + 재귀** — 각 fact를 트리플로 추가하고, 목적어를 새 엔티티로 등록한 뒤, 깊이 제한 이내이고 아직 방문하지 않았다면 확장 큐에 넣는다.
4. **통합** — 이미 방문한 엔티티는 건너뛰어 KB를 중복 제거한다.
5. **산출** — 엔티티를 확장할 때마다 스냅샷을 남겨 시드에서 바깥으로 KB가 자라는 과정을 볼 수 있게 하고, 최종적으로 `ontology.ttl`(OWL/Turtle, 트리플 구체화 포함), `ontology.json`(그래프), `steps.json`(단계별 스냅샷)을 기록한다.

## 4. 입력 / 출력
| 구분 | 파일 | 설명 |
|------|------|------|
| 입력 | `seed_entities.txt` | 시드 엔티티(한 줄에 하나, `#` 주석/빈 줄 무시). **코퍼스 없음.** |
| 출력 | `ontology.ttl` | 구체화된 KB (OWL/Turtle) |
| 출력 | `ontology.json` | Cytoscape 노드/엣지 (시각화용) |
| 출력 | `steps.json` | 엔티티별 스냅샷 — 재귀 크롤링 단계 재생용 |

## 5. LLM 백엔드
- 기본 `mock`: 키 없이 결정론적으로 동작. 작은 내장 지식 테이블이 각 "X에 대해 알려줘" 질의에 답하므로 재귀 확장과 골든 파일이 안정적이다.
- `api`(gemini/anthropic): env에 키가 있으면 실제 LLM이 자신의 파라미터 지식을 구체화한다(훨씬 크고 풍부). 키가 없으면 자동으로 MOCK으로 폴백한다.

## 6. 직접 해보기
1. `samples/seed_entities.txt`를 편집한다(예: `Motor`, `Pump`).
2. 사이트의 **Run** 버튼을 누른다(또는 `python pipeline.py samples runs/out`).
3. 스텝 슬라이더로 KB가 시드 → depth 1 → depth 2로 확장되는 과정을 본다.
